# 五洲 SaaS 客户交付：SOP 与上线流程

本文档面向「将五洲 SaaS 交付给客户使用」的场景，包含：**系统核心概念**、**标准作业流程（SOP）** 与 **上线流程介绍**，供实施/交付团队与客户共同遵循。文档为使用说明，面向非技术岗位人员。

---

## 一、系统核心概念

### 1.1 系统定位与主流程

五洲 SaaS 是面向**国际贸易/进口商品**的 B2B 管理系统，主业务流程为：

**采购 → 入库 → 销售 → 出库 → 结算**

### 1.2 重要板块说明

系统由以下**五大板块**组成，部分板块下包含子功能：

| 板块 | 说明 | 包含的子功能/管理内容 |
| --- | --- | --- |
| **供应链** | 采购从供应商拿货的全过程 | 供应商管理、供应链订单、供应链发票、货运信息等 |
| **销售** | 向商户（客户）销售的全过程 | 商户（商家）管理、销售订单、销售退货、销售发票、货运信息等 |
| **仓库** | 实物进出与库存管理 | 仓库、入库、出库、退库、盘点、移库等 |
| **商品** | 商品主数据与库存视图 | 分类、品牌、商品（SPU/SKU）、价格、库存查询等 |
| **财务** | 收付款分类与对账结算 | 财务类型、供应链应付/销售应收、税票等 |

- **供应商管理**归属在**供应链**板块内。
- **商户（商家）管理**归属在**销售**板块内。

各板块配合使用：先维护好**商品**与**供应链/销售**中的主数据（供应商、商户），再通过**供应链订单**采购、**仓库**入库，通过**销售订单**销售、**仓库**出库，最后在**财务**板块完成对账与结算。

---

## 二、各板块作用与完整操作流程

### 2.1 商品板块

**作用**：维护商品主数据（分类、品牌、商品 SPU/SKU、成本价/销售价等），为供应链与销售提供选品与定价基础。库存数据也在此板块查询。

**典型操作**：新建/维护分类与品牌 → 新建/维护商品及 SKU → 维护采购价、销售价、折扣价等；日常可查询各仓库、各 SKU 的库存。

---

### 2.2 供应链板块（含供应商管理）

**作用**：管理供应商档案，完成从下单、发货到入库、付款的完整采购流程。

**完整操作流程**：

1. **供应商管理**  
   在供应链板块下维护**供应商**档案（名称、联系人、结算方式等），便于下单时选择。

2. **创建供应链订单**

   - 选择**供应商**。
   - **添加商品**：选择 SKU、数量，维护各商品的**采购价**。
   - 保存订单（通常为草稿或待确认状态）。

3. **提审与审批**  
   提交审批；审批通过后订单进入执行阶段。

4. **维护货运信息**  
   拿到实际货运信息后，在订单下**补充货运信息**（如船名/航次、柜号、预计到港日等）。  
   若一批货对应订单中的部分商品，需**选择该货运关联的商品**，保证到货与入库能对应到具体商品行。

5. **入库**  
   货物到仓后，在**仓库**板块选择该供应链订单（及对应货运），为到货商品办理**入库**；入库时可维护**货架位**、**保质期（过期日）** 等（参见 2.4 仓库板块）。

6. **供应链财务处理**  
   在**财务**板块处理该供应链订单的应付：登记**供应链发票**（付款记录），选择**财务类型**、金额、付款时间等，直至该订单应付结清。

7. **结算**  
   在供应链侧完成该订单的**结算**。  
   **注意**：结算会用到商品的**体积数据**，用于**平摊该批次的运费成本**到各商品。请确保商品主数据中已维护好体积，否则运费分摊可能不准确。

---

### 2.3 销售板块（含商户管理）

**作用**：管理下游客户（商户），完成从下单、发货到收款、结算的完整销售流程。

**完整操作流程**：

1. **商户（商家）管理**  
   在销售板块下维护**商户**档案（名称、地址、结算方式等），便于下单时选择。

2. **创建销售订单**

   - 选择**商户**。
   - **添加商品**：选择 SKU、数量，维护各商品的**售价**（及折扣价如有）。
   - 保存订单（通常为草稿或待确认状态）。

3. **提审与审批**  
   提交审批；审批通过后订单可进入发货与出库。

4. **发货与货运信息**  
   实际发货后，在订单下**维护货运信息**（快递/物流单号、发货时间等）。

5. **出库**  
   发货会涉及**出库**：在**仓库**板块根据销售订单办理**出库**，扣减对应仓库、对应货架/批次的库存（参见 2.4 仓库板块）。

6. **销售财务处理**  
   在**财务**板块处理该销售订单的应收：登记**销售发票**（收款记录），选择**财务类型**、金额、收款时间等，直至该订单应收结清。

7. **结算**  
   在销售侧完成该订单的**结算**。  
   **注意**：销售结算时，系统会依赖**商品所对应的供应链订单是否已完成结算**。只有该销售订单所涉及商品所属的**供应链订单均已结算**，才能准确计算**周转率、年化收益**等指标。因此建议先完成相关供应链的结算，再做销售结算。

---

### 2.4 仓库板块（出入库与货架、库存拆分）

**作用**：管理仓库档案，办理采购入库、销售出库、退货退库，以及盘点、移库；系统按规则将库存拆分为多条可管理的「库存记录」。

**出入库与货架**：  
办理**入库**时，可填写**货架位**、**保质期（过期日）**；办理**出库**时，可选择从哪个货架/哪批库存出货。**盘点**、**移库**也会按货架与批次操作。

**库存拆分规则**（重要）：  
系统不会把同一 SKU 在同一仓库的所有数量混成一条库存，而是按以下**五个维度**拆分。仅当这五者**全部相同**时，才会合并为同一条库存记录；否则会新建或维护为**独立的库存记录**：

| 维度           | 说明                                   |
| -------------- | -------------------------------------- |
| 商品 SKU       | 具体规格型号                           |
| 所在仓库       | 哪个仓库                               |
| 过期日         | 保质期/过期日期（若无则视为同一类）    |
| 所属供应链订单 | 来自哪笔采购订单（若无则按无订单批次） |
| 货架位         | 存放的货架位置                         |

这样便于按批次、按货架管理库存，并支持先进先出、保质期管理等。

**典型操作**：

- **入库**：选择供应链订单（及货运）、仓库，选择要入库的商品与数量，填写货架位、过期日，提交入库。
- **出库**：选择销售订单、仓库，选择要出库的商品与数量（可从具体货架/批次选择），提交出库。
- **退库**：销售退货时，在仓库板块办理退库，选择退货单与商品、数量，退回指定仓库（可带货架位、过期日）。
- **盘点**：对指定仓库、货架或 SKU 进行盘点，录入实盘数量，系统据此调整库存。
- **移库**：在同一仓库内或跨仓库调整货架位/仓库，选择源库存与数量、目标货架/仓库，提交移库。

---

### 2.5 财务板块（含财务类型创建）

**作用**：对收付款方式进行分类（财务类型），并在此基础上登记供应链应付、销售应收、税票等，支持对账与结算。

**财务类型的创建与使用**：

- **财务类型**用于区分不同的**支付/收款方式**，例如：银行转账、现金、支票、承兑、线上支付等。
- **创建方式**：在财务板块进入「财务类型」管理，新增一条类型，填写名称（如「银行转账」）、是否默认等，保存即可。
- **使用场景**：在登记**供应链发票**（付款）或**销售发票**（收款）时，每笔发票需选择一种**财务类型**，便于按支付方式统计与对账。

建议上线前或业务需要时，先创建好常用财务类型，再在应付/应收中选用。

## 三、日常操作规范与异常处理

### 3.1 日常规范

- **状态流转**：供应链订单、销售订单均需按系统规定的状态顺序操作（如草稿 → 待审 → 已审 → 发货/入库 → 结算），不要跨状态操作。
- **数据一致**：入库、出库数量需与订单一致；财务登记金额、时间需与真实收付款一致。
- **权限**：按角色分配各板块权限，避免越权；管理员定期复核账号与权限。

### 3.2 异常与回退

- **误创建或误审**：若业务允许，通过系统提供的「作废」或「删除」处理，并确认是否影响库存或财务，必要时联系管理员。
- **库存异常**：通过**盘点**校正；重大差异需线下记录原因，再在系统中做盘点单。
- **系统报错或无法操作**：记录操作步骤与页面提示，联系实施方或技术支持。

## 四、上线流程介绍

### 4.1 上线前准备

- **系统访问**：确认客户能正常打开系统地址并登录（如需，由实施方配置网络或权限）。
- **账号与权限**：为客户创建至少一个管理员账号，并根据组织架构创建采购、仓库、销售、财务等角色，分配对应菜单与操作权限。
- **基础数据**：
  - 系统级：如省份、**财务类型**（参见 2.5）。
  - 业务级：仓库、**供应商**、**商户**、商品分类与品牌。
  - 商品：首批商品及 SKU、采购价/销售价、**体积**（供应链结算运费分摊需要）。

若有旧系统数据，需提前约定由谁在何时完成导入或手工录入。

### 4.2 上线步骤与检查项

| 阶段 | 步骤概要 | 检查项 |
| --- | --- | --- |
| 环境与账号 | 系统可访问，管理员及业务角色已创建并分配权限 | 各角色能登录且只能看到授权功能 |
| 基础数据 | 仓库、供应商、商户、分类品牌、首批商品、财务类型就绪 | 列表与详情正确，商品含体积等结算所需字段 |
| 流程验证 | 按「供应链：下单→审批→货运→入库→财务→结算」及「销售：下单→审批→发货→出库→财务→结算」各跑通至少一条 | 状态正确、库存变化正确、结算结果符合预期 |
| 培训与交接 | 对关键用户做系统与 SOP 培训，交付本 SOP 及操作手册 | 客户能独立完成主流程与日常操作 |

### 4.3 验收与交接

- **验收标准**：以合同或《验收清单》为准，一般包括系统可用、账号权限正确、主流程跑通、基础数据就绪、培训完成。
- **交接物**：管理员账号及权限说明、系统访问方式、本 SOP 与操作手册、支持联系方式与问题反馈流程。

### 4.4 上线后支持

- 客户通过约定渠道反馈问题，建议提供：操作人角色、操作步骤、页面或提示信息、截图。
- 版本或功能变更由实施方提前通知；涉及结算、库存的变更会特别说明。
- 按约定进行数据备份与权限复核。
